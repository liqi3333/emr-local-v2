## 一、需求回顾

### 1.1 左侧疾病树管理

在左侧疾病栏实现管理插件，支持：
- 添加一级分类（如"呼吸科"、"心血管科"）
- 添加二级疾病（如"肺炎"、"肺癌"）
- 删除现有疾病和分类
- 修改疾病和分类名称
- 修改分类的图标、颜色样式

### 1.2 新疾病的功能完整性

新添加的疾病必须能够使用所有现有病历功能（首次病程录、主治查房、主任查房、术前小结、术前讨论、手术记录、出院小结、各类同意书、护理记录等）。

### 1.3 右侧字段管理

右侧字段管理为已有功能（访问 `/record-types`），本次不改动。

## 二、核心设计决策（已确认）

| 决策项 | 选择 | 说明 |
|--------|------|------|
| 引入唯一 ID | ✅ 是 | 分类和疾病均带 `id`，ID 仅用于目录管理 |
| 疾病名是否允许重复 | ❌ 不允许 | 全局疾病名唯一校验 |
| 导入/导出功能 | ✅ 需要 | 管理页面提供 JSON 导入/导出 |
| 排序方式 | 上移/下移按钮 | 与现有 RecordTypeManager 一致 |
| 离线 mock 优化 | ❌ 不需要 | 不改动 `templates.js` |
| 重命名时迁移历史记录 | ✅ 方案 A（同步更新） | 重命名时在事务中更新 `records.disease` |
| 二级疾病图标 | ❌ 不需要 | 疾病只显示名字，保持侧边栏简洁 |
| 新增分类样式来源 | 预设调色板 + emoji 列表 | 管理页面提供 12 色系 + 20 emoji 供选 |
| 颜色字段配置方式 | 简化为 1 个主色，自动派生 | UI 只暴露色系选择，数据层保留 3 字段 |

### 关键边界约定

> **疾病 ID 只在目录管理层使用；疾病名字符串仍是传给 AI、存入病历记录、显示在历史记录中的值。**

这条边界是整个方案的安全基石，确保现有 AI 生成、提示词模板、病历保存、聊天、导出等功能完全不受影响。

## 三、数据模型

### 3.1 疾病目录结构

```javascript
[
  {
    id: 'cat-uuid-001',       // 分类唯一 ID（UUID）
    name: '腹部急症',          // 分类显示名
    icon: '🔥',                // 图标（emoji 或单字符）
    color: '#fee2e2',          // 标记色
    textColor: '#b91c1c',      // 文字色
    bgColor: '#fef2f2',        // 背景色
    sortOrder: 0,              // 分类排序序号
    diseases: [
      { id: 'dis-uuid-001', name: '急性阑尾炎', sortOrder: 0 },
      { id: 'dis-uuid-002', name: '急性胆囊炎', sortOrder: 1 }
    ]
  }
]
```

### 3.2 ID 策略

- 默认目录的 ID 在 `src/data/diseaseCategories.js` 中硬编码（UUID）
- 用户新增的分类/疾病使用 `crypto.randomUUID()` 生成
- 重置操作恢复硬编码的默认目录

### 3.3 存储方式

- 存在 SQLite `settings` 表，`key = 'disease_categories'`，value 为 JSON 字符串
- 与 `record_registry` 同模式
- 启动时调用 `ensureDefaultDiseaseCategories()`，无数据则写入默认目录

### 3.4 样式自定义设计

- **数据模型保留 3 个颜色字段**（`color`/`textColor`/`bgColor`），与现有 10 个分类完全兼容，无需迁移

### 4.3 API 设计

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/diseases` | 获取完整疾病目录 |
| PUT | `/api/diseases` | 替换完整疾病目录（用于导入） |
| POST | `/api/diseases/category` | 添加一级分类 |
| PUT | `/api/diseases/category/:id` | 修改分类（名称/图标/颜色/排序） |
| DELETE | `/api/diseases/category/:id` | 删除分类及其下所有疾病 |
| POST | `/api/diseases/category/:id/disease` | 添加疾病 |
| PUT | `/api/diseases/category/:id/disease/:diseaseId` | 修改疾病（含重命名迁移） |
| DELETE | `/api/diseases/category/:id/disease/:diseaseId` | 删除疾病 |
| POST | `/api/diseases/reset` | 重置为默认目录 |

### 4.4 校验规则

- 分类名非空、全局不重复
- 疾病名非空、**全局不重复**（跨分类校验）
- 分类 ID、疾病 ID 唯一
- 图标允许 emoji 或单字符
- 导入时校验数据格式和唯一性，**校验失败拒绝整个导入**，返回具体错误信息

### 4.5 重命名迁移逻辑（方案 A）

`diseaseRegistry.js` 中实现：

```javascript
function renameDisease(catId, diseaseId, newName) {
  // 1. 校验新名字全局唯一（排除自身）
  // 2. 找到旧名字
  // 3. 在事务中：
  //    a. 更新目录中的疾病名
  //    b. 更新所有历史记录：UPDATE records SET disease = ? WHERE disease = ?
  // 4. 保存目录
}
```

使用 SQLite 事务保证目录和记录的原子性更新。

### 4.6 路由注册顺序

`/diseases` 页面路由必须在 `server.js` 的 SPA fallback (`app.get('*')`) 之前注册，与 `/prompts`、`/record-types` 同位置。

### 6.1 无影响的部分 ✅

| 功能 | 原因 |
|------|------|
| AI 在线生成 | 只接收 `disease` 字符串，不涉及 ID |
| AI 提示词模板 | `{{disease}}` 仍替换为名字 |
| 病历记录保存/读取 | `records.disease` 仍存字符串 |
| 聊天功能 | 使用 `currentDisease` 字符串 |
| Markdown 导出 | 使用当前选中的疾病名字 |
| 打印功能 | 与目录数据解耦 |
| 右侧 Registry/字段系统 | 完全独立模块 |
| 历史记录显示 | `PatientManager`、`EmrPreview` 显示 `r.disease`，不受影响 |

### 6.2 需要小改的部分 ⚠️

| 文件 | 改动 | 说明 |
|------|------|------|
| `public/js/data/diseases.js` | 改成 loader | 原常量改为动态获取 + 兜底 |
| `public/js/components/DiseaseTree.js` | 适配对象结构 | 疾病从字符串变为 `{id,name}` |
| `public/js/store.js` | 新增 `diseaseCategories` + `reset` 保留 | 支持目录变化时树自动刷新 |
| `public/index.html` | 标题 + 按钮 | 小改动 |
| `public/js/app.js` | 加载目录 + 跳转 + 同步 | 小改动 |

### 6.3 需要特别注意的点 ⚠️

1. **`store.reset()` 必须保留 `diseaseCategories`**：否则切换患者时疾病树会消失
2. **DiseaseTree 搜索重渲染**：数据从 store 读取，避免搜索时触发异步加载竞态
3. **`_updateActiveClass()` 用 `data-*` 属性匹配**：避免搜索时 `<mark>` 标签干扰 textContent 匹配
4. **`/diseases` 路由注册顺序**：必须在 SPA fallback 之前
5. **重命名同步更新历史记录**（方案 A）：在事务中执行 `UPDATE records SET disease = ? WHERE disease = ?`
6. **统计页面**：重命名后统计不再分裂（因为记录已同步更新）
7. **离线模板**：重命名后旧名模板失效，在线模式不影响；新疾病无专属模板，fallback 到通用 mock

## 十、给非技术人员的通俗说明

### 这个插件要解决什么问题？

打个比方，现在的系统就像一个**固定的菜单**：左侧列着固定的科室和疾病（比如"腹部急症"下面有"急性阑尾炎"、"急性胆囊炎"等），这些疾病是写死在代码里的。如果你想加一个新疾病，比如"呼吸科"下的"肺炎"，必须找程序员改代码，非常麻烦。

这个插件就是做一个**疾病管理后台**，让你像管理手机通讯录一样，自己就能增删改疾病目录，不用碰代码。

### 改了之后会影响什么？

**好消息是：大部分功能完全不受影响。**

- **写病历**：AI 生成病历时，只需要知道"疾病叫什么名字"。不管这个疾病是你新加的还是原有的，AI 都能正常工作。
- **聊天**：和 AI 对话时提到的疾病名也不变。
- **导出病历**：导出的文件里显示的疾病名照旧。
- **历史记录**：之前保存的病历不受影响。

**需要小调整的地方：**

- 左侧疾病栏从"写死的"变成"从数据库读的"，这样你改了之后能立刻看到变化。
- 疾病的存储方式加了一个"身份证号"（ID），就像每个人都有身份证号一样。这样即使你改了疾病的名字，系统也知道"这还是同一个疾病"，不会搞混。

### 几个关键设计用大白话解释

**1. 为什么要给疾病加 ID（身份证号）？**

假设你把"急性阑尾炎"改名叫"阑尾炎"。如果没有 ID，系统只认名字，就会以为这是一个新疾病，之前所有"急性阑尾炎"的病历就找不到对应疾病了。有了 ID，系统知道"ID 还是那个 ID，只是名字变了"，就能把所有历史记录的名字也一起更新过来。

**2. 为什么疾病名不能重复？**

如果"呼吸科"和"肿瘤科"下面都有一个叫"肺癌"的疾病，你点"肺癌"时系统不知道你说的是哪个。所以规定整个系统里疾病名必须唯一，就像一个班里不能有两个同名的同学（不然老师喊名字不知道叫谁）。

**3. 为什么要"同步更新历史记录"（方案 A）？**

你把"急性阑尾炎"改名叫"阑尾炎"后，如果历史记录还写着"急性阑尾炎"，会出现两个问题：
- 统计页面会把"急性阑尾炎"和"阑尾炎"算成两个不同的疾病，数字对不上
- 历史记录里显示的还是旧名字，看起来不一致

所以选择方案 A：改名字的时候，自动把所有历史记录里的旧名字也一起改成新名字。这样数据始终是一致的。这个操作在"事务"里完成——要么全部成功，要么全部不变，不会出现改了一半的尴尬情况。

**4. 为什么要"导入/导出"？**

你辛苦配置了一套自己的疾病目录，万一电脑重装或数据库坏了就全没了。导出功能可以把它存成一个文件备份；换电脑时用导入功能恢复。就像手机通讯录可以导出成文件再导入新手机一样。

**5. "兜底数据"是什么意思？**

万一读数据库的时候出了问题（虽然本地数据库一般不会出问题），系统不会直接崩溃，而是用一份备用的小数据集先显示出来，保证你能继续用。就像停电时手电筒自动亮起来应急。

**6. 分类样式（图标和颜色）怎么自定义？**

每个一级分类（比如"腹部急症"）都有图标和颜色。在管理页面里：
- **图标**：从一个表情符号列表里选（比如 🔥🫁❤️🚑），点哪个就显示哪个
- **颜色**：从 12 种预设色系里选一个（红、橙、黄、绿、蓝、紫……），选完之后系统自动帮你配好"标题色、文字色、背景色"三个颜色，不用你自己调色

这就像选手机主题一样——你只需要选"红色系"或"蓝色系"，剩下的系统帮你搭配好，保证看起来协调。

**二级疾病（比如"急性阑尾炎"）不带图标和颜色**，只显示纯文字名字，保持侧边栏干净简洁。

### 总结一句话

> 这个插件让你能自己管理疾病目录（增删改 + 自定义图标颜色），同时通过"ID + 同步更新历史记录"的设计，保证改了之后所有功能依然正常、数据保持一致，不会出现对不上账的情况。
