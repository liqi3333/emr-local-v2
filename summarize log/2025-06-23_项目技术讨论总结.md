# 项目技术讨论总结

> 日期：2025-06-23
> 项目：emr-local-v2（本地病历管理 + AI 生成系统）

---

## 1. 项目使用什么编程语言

**全栈 JavaScript**：
- 后端：Node.js + Express，CommonJS 规范（`require()`）
- 前端：原生 JS ES Modules，无框架、无构建、无打包器
- 数据库：SQLite（`better-sqlite3`）
- 样式：纯 CSS（CSS custom properties）
- 无 TypeScript、无测试套件、无 linter

---

## 2. JavaScript 的优缺点

### 优点
- 全栈同语言，上下文切换成本低
- Node.js 异步非阻塞，适合 IO 密集型（查 SQLite + 调 AI API）
- SSE 流式输出（AI 聊天）实现自然
- 零构建/零编译，改完代码刷新浏览器即生效
- JSON 是母语，与 Registry 配置、数据库 content 列天然契合
- npm 生态成熟

### 缺点
- **无类型检查**：`_buildRecordContent` 漏字段不报错，`typeId` 拼错不报错，全靠人记
- **运行时错误频繁**：`Cannot read properties of undefined` 常见
- **单线程**：同步计算会卡住事件循环
- **better-sqlite3 需要本地编译**（Windows 需额外环境）
- **异步易吞错误**：SSE 流式处理需额外小心

---

## 3. 操作系统的兼容性

### 开发阶段
| 系统 | 兼容情况 |
|------|---------|
| macOS / Linux | 开箱即用，命令兼容好 |
| Windows | better-sqlite3 需 C++ 编译环境；Unix 命令（lsof、cp）不支持，建议用 Git Bash 或 WSL2 |

### 实际使用阶段
- B/S 架构，用户仅需浏览器，跨平台一致
- SQLite .db 文件三平台直接拷贝使用
- 建议提供 `start.sh`（macOS/Linux）和 `start.bat`（Windows）一键启动脚本
- 可用 `pkg` 打包成各平台可执行文件，用户无需安装 Node.js

---

## 4. TypeScript 迁移建议（三个选项）

### 选项一：只改后端，前端不动（推荐先做）
- `src/` 下核心文件（database.js、recordRegistry.js、api.js 等）改为 .ts
- 用 `tsx watch` 运行，保留热重载
- 前端仍是纯 JS，零构建不变
- **收益最大、破坏最小**，解决后端漏字段/拼错问题

### 选项二：前后端都改 TS，用 esbuild 编译前端
- 前后端全改为 .ts
- esbuild 编译前端（~0.2s），刷新浏览器即见
- 启动时同时跑 `esbuild --watch` + `tsx watch`

### 选项三：前后端都改 TS，引入 Vite 开发服务器
- Vite 自动热更新，改代码浏览器自动刷新
- 需要调整项目结构，破坏"原生无构建"的简洁性
- 适合计划大幅扩展前端功能的场景

---

## 5. 从头构建类似项目的技术选型

**推荐：TypeScript（严格模式）+ Node.js + esbuild（前端轻编译）+ Drizzle ORM + SQLite**

### 为什么会选这套组合
- 类型安全：消灭运行时静默错误（漏字段、拼错 ID）
- 仅增加 ~200ms 前端编译延迟，保留快速迭代体验
- 已有 JS 基础，迁移成本最低

### 数据库选型理由

**SQLite 仍是最优选择**：
- 部署成本为零（无服务、无配置、单文件）
- 跨平台极致兼容
- 当前数据量（单医生/科室）完全在能力范围内
- 不温不热

**但结构和访问层要升级**：不再使用独立的 `content` JSON 列做泛型存储，而是每个病历类型一张表，用 Drizzle ORM 自动推导类型，编译时保证列不遗漏。

### 为什么不选其他语言
| 语言 | 理由 |
|------|------|
| Go | 前端仍是 JS/TS，要维护两个语言 |
| Python | 类型浮于表面，打包分发不如 Node 的 pkg 成熟 |
| Rust | 开发速度慢一个数量级，学习曲线陡峭 |

---

## 6. 核心结论

1. JS 让项目**快速成型**，但**质量全靠开发者纪律**。
2. SQLite 对于本地病历系统是**正确的数据库选择**。
3. TypeScript + 后端的**渐进式迁移**是最优路径——先改后端核心文件，获得最大的安全性提升，同时保留前端零构建的便捷。
