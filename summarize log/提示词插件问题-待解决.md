# 提示词插件问题-待解决

## 背景

病历系统重构为双层标签（分类→类型）后，提示词编辑插件需要同步重构。原先只对应 7 个老类型，现在需覆盖全部 13 个类型，并支持未来动态增删字段。

---

## 已确认的设计决策

经过讨论，确定以下方向：

| 决策项 | 选择 |
|--------|------|
| 字段描述唯一权威 | **Registry** 为唯一真实来源，`defaultPrompts.json` 只保留 rolePrompt/outputFormat/endingPrompt/userPrompt |
| 新增类型（generic_xxx）是否可编辑完整提示词 | **是**，PromptEditor 需支持所有 Registry 类型 |
| 上下文占位符命名风格 | **跟随 storeKey**，例如 `{{emrDataContext}}`、`{{nursingAssessmentDataContext}}` |
| ChatArea 是否接入 PromptEditor 模板 | **是**，聊天修改病历时继承当前模板的 rolePrompt 和字段规范，但保留"最小改动"的聊天专属约束 |
| defaultPrompts.json 详细字段描述 | **合并进 Registry**，然后清空 defaultPrompts.json 中的字段 description |
| 类型与模板联动 | **新增类型自动生成模板骨架，删除类型自动清理模板覆盖**（选项 B） |
| templateKey 修改策略 | **已有类型在配置器中只读，不开放修改**（防止破坏模板绑定关系） |

---

## 问题清单

### 问题 1：编辑器只显示 7 个类型（硬编码）

**文件：** `public/js/components/PromptEditor.js:14`

```javascript
const TAB_ORDER = ['emr', 'attending', 'chief', 'preop', 'discussion', 'surgery', 'discharge'];
```

新增的 6 个同意书/护理模板（`surgeryConsent`、`bloodTransfusionConsent`、`anesthesiaConsent`、`nursingAssessment`、`nursingPlan`、`nursingRecordSheet`）在编辑器中完全不可见，用户无法编辑它们的提示词。

---

### 问题 2：编辑器与 Registry 没有联动

编辑器读 `defaultPrompts.json` 的 key 列表，Registry 的类型配置对编辑器不可见。用户在配置器中新增/删除/重命名类型后，编辑器不会自动更新。

**理想行为：** 编辑器应从 Registry 动态读取所有类型（包括新增的），而不是硬编码 TAB_ORDER。

---

### 问题 3：58/88 个字段描述不一致

Registry 中的描述是**简短版**，模板中的描述是**详细版**。两套数据各说各话。

| 类型 | 字段数 | 描述一致 | 描述不一致 |
|------|--------|---------|-----------|
| 7 个老类型 | 46 | 24 | **22** |
| 6 个新类型 | 42 | 8 | **34** |

**示例：** `chief`（初步诊断）
- Registry：`'初步诊断'`
- 模板：`'初步诊断。**既往史中写到的疾病，诊断中必须有对应条目**，不可遗漏...'`

**核心矛盾：** 两套数据源，哪套是权威？
- `recordRegistry.js` 的 `fields[].description` — 简短版，渲染/AI 聊天/Mock 的数据源
- `defaultPrompts.json` 的 `fields[key].description` — 详细版，AI 提示词的数据源

**决策：** Registry 为唯一权威，将 `defaultPrompts.json` 中更详细的描述合并进 Registry，模板中的 `fields[key].description` 废弃或清空。

---

### 问题 4：上下文依赖占位符缺失

**文件：** `src/services/promptTemplates.js:338-357`

`replacePlaceholders()` 只支持 6 个占位符：
```
{{disease}} {{patientContext}} {{emrContext}}
{{attendingContext}} {{preopContext}} {{surgeryContext}}
```

但 Registry 中的上下文依赖关系：
- `nursingPlan` 依赖 `nursingAssessment`
- `nursingRecordSheet` 依赖 `nursingPlan`

API 加载了这些依赖数据，但提示词中**没有对应的占位符**（如 `{{nursingAssessmentContext}}`、`{{nursingPlanContext}}`），数据无法注入。护理评估的内容无法传递给护理计划。

**决策：** 占位符统一跟随 storeKey，全部改为 `{{xxxDataContext}}` 格式，并补齐护理类型占位符。

---

### 问题 5：配置器新增类型的 templateKey 无法进入编辑器

新增类型的 `templateKey` 是 `generic_xxx`，`defaultPrompts.json` 中没有这个 key，编辑器也不会显示它。这些类型只能走 Layer 2 自动生成提示词（`buildFromRegistryFields()`），用户无法在编辑器中自定义。

**决策：** PromptEditor 从 Registry 读取所有类型，对没有默认模板的类型动态生成可编辑模板骨架。

---

### 问题 6：Diff 检测不完整

**文件：** `public/js/components/PromptEditor.js:653-673`

`_showDiff()` 只检测：
- `[新增字段]` — 字段存在于 default 但不在 current 中
- `[新增类型]` — 类型存在于 default 但不在 current 中

不检测：
- rolePrompt、outputFormat、endingPrompt、userPrompt 的内容修改
- 字段 description 的修改
- 字段被删除的情况

用户修改了提示词主体内容后点"查看差异"，可能显示"当前模板与默认模板一致"。

---

### 问题 7：重置功能 UX 问题

#### 字段重置（`_resetFieldEdit()`，行 544-553）
只重置 UI 状态（`this._state.editingField.label/description`），不删除已保存的自定义覆盖。用户需手动点"保存字段"才生效。期间字段表格仍显示"自定义"标签，造成困惑。

#### 标签页重置（`_resetCurrentTab()`，行 612-628）
`delete template.templates[typeKey]` 删除覆盖但不保存。Toast 提示"记得点击保存"，但用户切换标签页或切换模板后，重置操作静默丢失。无脏状态追踪。

---

### 问题 8：PromptEditor 预览与实际发送的 system prompt 不一致

`_renderPreview()` 在浏览器里自己拼装预览，没调用后端的 `assembleSystemPrompt()`，也不做占位符替换。用户看到的预览和 AI 实际收到的内容可能不一致，尤其是 Layer 2 自动生成的类型。

---

### 问题 9：Layer 1 模板忽略 Registry 的字段变化

`assembleSystemPrompt()` 只要 `defaultPrompts.json` 里存在该类型，字段块就取自 `defaultPrompts.json`。这导致：
- Registry 新增字段 → AI 提示词里没有
- Registry 禁用字段 → AI 提示词里还有
- Registry 修改 label/description → AI 提示词不变

**修复：** Layer 1 只保留模板的 rolePrompt/outputFormat/endingPrompt/userPrompt，字段块必须从 Registry 读取。

---

### 问题 10：ChatArea 自己写死了一套系统提示词，和 PromptEditor 无关

`ChatArea.js` 的系统提示词是硬编码字符串，只取了 Registry 字段的 `label`，没读 PromptEditor 模板。用户在 PromptEditor 里修改 rolePrompt 或字段描述，对聊天修改病历功能完全不生效。

---

### 问题 11：Mock 生成丢弃上下文依赖

`mockGenerate()` 给 strategy 的 prompt 是合成的 `请为"xxx"生成...`，不带 `emrData`、`attendingData` 等依赖数据。主治查房、主任查房等 mock 无法继承首次病程录内容。

---

### 问题 12：模板版本号不跟踪 Registry 变化

`defaultPrompts.json` 的 `version` 不会随 Registry 字段变化而更新，导致：
- 更新 banner 不弹出
- `syncTemplate()` 不会同步 Registry 字段变化
- Diff 检测不到字段级变更

---

### 问题 13：新增/删除类型后提示词模板不同步

在病历类型管理插件新增一个类型后，提示词插件不会自动出现该类型的模板。
删除一个类型后，自定义模板中可能残留该类型的覆盖数据。

**决策：** 选择方案 B — 新增类型时自动生成模板骨架，删除类型时自动清理模板覆盖。
已有类型的 `templateKey` 在配置器中只读，不开放修改。

---

## 修复方向建议

1. **编辑器从 Registry 动态读取类型列表**，去掉硬编码 TAB_ORDER
2. **统一字段描述数据源**，Registry 成为唯一权威
3. **占位符统一为 `{{xxxDataContext}}`**，补齐护理类型上下文占位符
4. **支持新增类型的提示词自定义**，编辑器显示所有 Registry 类型并动态生成模板骨架
5. **完善 Diff 检测**，比较 rolePrompt/outputFormat/endingPrompt/userPrompt/字段 label/字段 description/字段删除
6. **修复重置 UX**，字段重置立即生效，标签页重置加确认框并保存
7. **PromptEditor 预览真实化**，预览内容等于 AI 实际收到的 system prompt
8. **assembleSystemPrompt 字段块从 Registry 读取**，跟随字段启用/禁用/新增/删除
9. **ChatArea 接入 PromptEditor 模板**，聊天系统提示词使用当前模板 rolePrompt + Registry 字段说明
10. **Mock 生成传入 context**，strategy 从 context 读取依赖数据
11. **模板版本跟踪 Registry 变化**，Diff/Sync 能感知 Registry 字段变更
12. **新增/删除类型时自动同步模板**，配置器操作直接反映到提示词插件
13. **已有类型 templateKey 只读**，防止破坏模板绑定关系

---

### 阶段八：类型生命周期与模板同步

**目标：** 新增/删除类型时，提示词模板自动同步，避免残留垃圾数据。

#### 子阶段 8.1：新增类型自动生成模板骨架（30-45 分钟）

- 在 `src/services/promptTemplates.js` 新增 `ensureTypeTemplateSkeleton(templateKey, registryTypeConfig)`。
- 生成默认 `rolePrompt`（如 `"你是一位普外科责任护士，请根据疾病...`）、`outputFormat`、`endingPrompt`、`userPrompt`。
- 字段列表从 Registry 读取，description 使用 Registry 值。
- 在 `src/routes/recordTypes.js` 的 `POST /category/:id/type` 中调用：
  - 如果 `defaultPrompts.json` 中已有该 `templateKey`，跳过。
  - 如果当前激活的自定义模板中已有该 `templateKey`，跳过。
  - 否则，在激活模板中动态生成骨架。
- 验证：在配置器新增一个类型后，打开 `/prompts` 能看到并编辑该类型。

#### 子阶段 8.2：删除类型自动清理模板覆盖（30-45 分钟）

- 在 `src/services/promptTemplates.js` 新增 `removeTypeTemplateOverrides(templateKey)`。
- 遍历 `data/prompt-templates/*.json`，删除每个自定义模板中 `templates[templateKey]` 的覆盖数据。
- 不删除 `defaultPrompts.json` 中的默认模板（只读）。
- 在 `src/routes/recordTypes.js` 的 `DELETE /category/:id/type/:typeId` 中调用。
- 验证：删除一个类型后，自定义模板中该类型的覆盖被清理。

#### 子阶段 8.3：Registry 重置可选同步清理模板（15-30 分钟）

- `POST /record-types/reset` 支持可选参数 `resetTemplates`。
- 为 `true` 时，将所有自定义模板的 `templates` 中已不存在的类型覆盖清理掉。
- 为 `false` 或不传时，只重置 Registry，不动模板。
- 验证：重置 Registry 后，自定义模板中残留的类型覆盖被清理。

#### 子阶段 8.4：前端提示与确认（15-30 分钟）

- `RecordTypeManager.js`：
  - 新增类型成功后，toast 提示："已自动生成提示词模板，可在 /prompts 中编辑"。
  - 删除类型时，弹出确认框："此操作会同步删除该类型在提示词插件中的自定义配置，是否继续？"
- 验证：新增/删除类型时，提示信息正确，用户可选择取消。

#### 子阶段 8.5：templateKey 只读限制（15-30 分钟）
