# 病历插件系统 — 程序员部分深度分析

## 问题 1：60 列扁平 Schema

### 根源

早期设计为每种病历类型预留独立列，随着类型从 7 增加到 13（88 个字段），`records` 表有 77 个数据列。每条记录只用到 5-9 个列，其余全是空字符串。

### 相关代码

- `src/services/database.js:21-38` — `RECORD_DATA_COLUMNS` 数组（77 列）
- `src/services/database.js:69-125` — `CREATE TABLE records` 定义
- `src/services/database.js:133-171` — `ALTER TABLE` 动态补列
- `src/services/database.js:278-306` — `saveRecord()` 的 INSERT/UPDATE SQL

### 数据浪费实测

| 记录类型 | 使用列数 | 空列数 | 浪费比例 |
|----------|---------|--------|---------|
| chiefRound（最少） | 5 | 72 | 93.5% |
| firstCourse（最多） | 9 | 68 | 88.3% |

### 额外问题

- 7 个列在多个类型间共享（`surgeryName`、`alternatives`、`patientSignature`、`consentDate`、`patientCondition`、`vitalSigns`、`nursingDiagnosis`），同患者的不同记录会互相覆盖扁平列的值
- `content` JSON 只在 INSERT 时写入，UPDATE 时不更新，导致两个表示之间存在静默数据漂移

### 解决步骤

**方案 A（推荐）：** `content` JSON 列作为主存储，扁平列降级为索引列

1. 修改 `saveRecord()` — INSERT 和 UPDATE 都写入 `content` JSON
2. 修改 `getRecords()` / `getRecordById()` — 从 `content` JSON 读取字段数据
3. 保留 `RECORD_DATA_COLUMNS` 用于向后兼容，标记为 DEPRECATED
4. 数据库迁移：已有数据的扁平列值迁移到 `content` JSON

**方案 B（次选）：** 保持扁平列，UPDATE 时同步写入 `content` JSON

1. 最小改动：`saveRecord()` 的 UPDATE 路径加上 `content = ?`
2. 不解决空间浪费，但解决数据漂移

### 解决后影响

- 方案 A：需要数据库迁移脚本，`getRecords()` 查询逻辑变化，前端无感知
- 方案 B：零破坏性，仅修复数据一致性

## 问题 2：Prompt 编辑器与 Registry 脱节

### 根源

编辑器已通过 `recordTypeApi.getRegistry()` 动态读取 Registry，但有隐藏门控：`PromptEditor.js:104` 要求 Registry 类型的 `templateKey` 必须在 `defaultPrompts.json` 中有对应条目才显示为 Tab。配置器新增的类型（`templateKey` 为 `generic_xxx`）不满足此条件。

### 相关代码

- `public/js/components/PromptEditor.js:86-114` — `_buildTabOrder()` 从 Registry 构建 Tab 列表
- `public/js/components/PromptEditor.js:104` — 门控条件：`if (this._state.defaultTemplate?.templates?.[type.templateKey])`
- `public/js/components/PromptEditor.js:699-711` — `_collectCurrentTabData()` 从 `defaultTemplate` 读取 label
- `public/js/components/PromptEditor.js:252-253` — `_renderTabContent()` 从 `defaultTemplate.templates[typeKey]` 读取字段
- `public/js/components/PromptEditor.js:792-811` — `_showDiff()` 只迭代 `defaultTmpl.templates`

### 解决步骤

1. 移除 `_buildTabOrder()` 第 104 行的门控条件，所有 `enabled !== false` 的 Registry 类型都显示为 Tab
2. `_renderTabContent()` 增加 fallback：当 `defaultTemplate.templates[typeKey]` 不存在时，从 Registry 的 `fields` 数组渲染字段表
3. `_collectCurrentTabData()` 的 `label` 来源增加 Registry fallback
4. `_showDiff()` 同时迭代 Registry 类型和 Default 模板类型

### 解决后影响

- 新增类型可以在编辑器中编辑提示词
- 编辑器完全由 Registry 驱动，不再依赖 `defaultPrompts.json` 的硬编码列表
- 向后兼容：13 个内置类型不受影响

## 问题 3：两套字段描述

### 根源

经调查发现**不是实际问题**。`defaultPrompts.json` 的字段对象只有 `label`，没有 `description` 属性。`assembleFieldBlock()` 中 `templateFields[key]?.description` 始终返回 `undefined`，自动 fallback 到 Registry 描述。

### 实际数据流

| 场景 | AI 看到的描述来源 |
|------|------------------|
| 默认模板，13 个内置类型 | Registry（模板无 description） |
| 自定义模板，字段未编辑 | Registry（合并后仍无 description） |
| 自定义模板，字段已编辑 | 用户自定义文本 |
| Layer 2 fallback（新类型） | Registry |
| ChatArea API fallback（网络故障） | **仅 label，丢失描述** |

### 真正的风险点

- 如果有人在 `defaultPrompts.json` 中给字段加了 `description`，会静默覆盖 Registry 描述
- `ChatArea.js:147-158` 的 fallback 路径丢弃所有描述，只发送 `key: label`，AI 输出质量严重下降

### 解决步骤

1. 在 `assembleFieldBlock()` 中反转优先级：Registry 描述优先，模板 description 仅作为补充
2. `ChatArea.js` 的 fallback 路径使用 Registry 描述而非仅 label
3. 在 `defaultPrompts.json` 的字段中永远不要添加 `description` 属性

### 解决后影响

- 消除描述覆盖风险
- ChatArea fallback 路径输出质量恢复

## 问题 4：上下文依赖占位符缺失

### 根源

经调查发现**不是实际问题**。`replacePlaceholders()` 使用动态生成机制：遍历 `context` 对象的所有 key，为每个非空对象生成 `{{storeKeyContext}}` 占位符。`CONTEXT_LABELS` 映射表已包含全部 13 个类型的 storeKey。

### 验证

- `nursingPlan` 的 `rolePrompt` 包含 `{{nursingAssessmentDataContext}}` ✅
- `nursingRecordSheet` 的 `rolePrompt` 包含 `{{nursingPlanDataContext}}` ✅
- 所有 13 个类型的依赖关系与占位符完全匹配 ✅

### 唯一风险

如果依赖数据为空（尚未生成），`replacePlaceholders` 的 `Object.keys(value).length > 0` 守卫会阻止替换，字面文本 `{{nursingAssessmentDataContext}}` 会出现在 AI 提示词中，可能混淆模型。

### 解决步骤

1. 在 `replacePlaceholders()` 中，对未替换的占位符替换为空字符串或注释说明（如"（护理评估尚未生成）"）
2. 或者在前端生成前检查依赖数据是否存在，缺失时给出提示

### 解决后影响

- 消除字面占位符泄露到 AI 提示词的风险

## 问题 5：无 undo/redo

### 根源

`contentEditable` 的浏览器原生 undo 依赖 DOM 元素的存活。但 `EmrPreview.js:98` 的 `container.innerHTML = ''` 在每次状态变化时销毁并重建整个 DOM，浏览器的 undo 栈随之丢失。

### 相关代码

- `public/js/components/EmrPreview.js:98` — `container.innerHTML = ''` 销毁 DOM
- `public/js/components/EmrPreview.js:159` — `contentEditable = 'true'`
- `public/js/components/EmrPreview.js:164-176` — blur/keydown 事件处理器
- `public/js/store.js` — 无 undo 栈机制

### 解决步骤

**完整方案：** 在 store 中维护 undo 栈，每次 `setTypeData()` 前记录快照，Ctrl+Z 时恢复上一个 snapshot

**简单方案：** 仅在编辑区域旁增加「重置为上次保存值」按钮，从 DB 读取原始数据恢复

### 解决后影响

- 完整方案：开发量中等（约 50-80 行），用户体验显著提升
- 简单方案：开发量小（约 20 行），但只能重置到保存时状态
