## 二、各任务详细说明

### P0：RAG 知识库系统

**目标**：为 AI 生成提供专业医学知识上下文，提升病历生成的领域准确性。

**后端实现**：
- 新建 `src/services/knowledge.js`
  - `loadKnowledge(disease)` — 按疾病名读取对应 MD 文件
  - 8000 字符上限截断，防止超出 LLM 上下文窗口
- 新建 `src/data/medical-files/` 目录
  - 按疾病名建子目录，存放 MD 格式专业医学文件
  - 示例：`腹股沟疝/诊断标准.md`、`腹股沟疝/手术指征.md`
- 改造 `src/routes/api.js`
  - 生成端点中注入知识库内容到 system prompt
  - 知识库内容位于 user prompt 之前，AI 优先基于文件内容生成
- 离线兼容：知识库在路由层加载，在线/离线模式均可使用

**前端实现**：
- 在 SettingsPanel 或独立页面增加知识库管理入口
- 文件上传界面：支持上传 MD 文件到指定疾病目录
- 已上传文件列表：查看、删除

**工作量**：小（后端约 80 行，前端约 150 行，UI 改造约 30 行）

---

### P1：模板端点合并

**目标**：将 6 个独立的模板查询端点合并为 1 个通用端点。

**当前状态**（6 个独立端点）：
```
GET /api/templates/attending/:disease
GET /api/templates/chief/:disease
GET /api/templates/preop/:disease
GET /api/templates/discussion/:disease
GET /api/templates/surgery/:disease
GET /api/templates/discharge/:disease
```

**合并后**（1 个通用端点）：
```
GET /api/templates/:type/:disease
```
其中 `:type` 取值：`attending` | `chief` | `preop` | `discussion` | `surgery` | `discharge`

**工作量**：小（后端约 20 行，前端约 30 行）

> ⚠️ **前置条件**：如果决定做"病历类型插件系统"，则此任务可直接并入插件系统的动态路由设计，无需单独实施。

---

### P2：智能模板进化系统

**目标**：系统根据已保存的真实病例自动学习，生成更贴近实际的病历模板。

**触发条件**：
- 某疾病已有 ≥3 份已保存病例时自动触发

**聚合方式**：
- AI 分析共同模式：主诉模式、体征特点、诊断模式、治疗方案
- 生成聚合模板，存入 `learned_templates` 表或 `src/data/learned-templates/` 目录

**模板使用**：
- 下次生成同疾病病例时，优先使用聚合模板
- 比预置模板更贴近实际临床书写习惯

**版本管理**：
- 每次聚合生成新版本
- 保留历史版本
- 支持用户编辑修正
- 支持回退到旧版本

**工作量**：中

---

### P2：小模型 + RAG 分级策略

**目标**：通过分级策略降低 AI 调用成本，同时保证生成质量。

**核心思路**：
```
有知识库的疾病 → 小模型（Ollama 1.5B–7B）→ 评分 80-85 分 ✅ 免费
无知识库的疾病 → 回退到大模型（GPT-4o/Claude）→ 评分 70 分      💰 付费
```

**依赖**：必须先完成 P0 RAG 知识库系统

**完全离线**：Ollama 本地运行 + 本地知识库，零网络依赖

**工作量**：中

**目标**：支持双层标签（文档种类 → 病历类型），可动态添加/删除/开关病历类型。

**4 个实施阶段**：

**工作量**：大（总计 9-13 天）

## 四、推荐策略

```
现在就去做的   → P0: RAG 知识库系统
做完 RAG 后   → P2: 小模型+RAG 分级策略（收益高，成本低）
有闲时间做    → P2: 智能模板进化系统
评估后再决定  → P3: 病历类型插件系统（工作量较大，需评估必要性）
```

### 理由

1. **RAG 知识库收益最高**：不改变现有架构，增量开发，完成后 AI 生成质量明显提升
2. **小模型+RAG 是最具性价比的方案**：充分利用本地 Ollama 的能力，减少对大模型的依赖
3. **智能模板进化系统**：属于锦上添花，数据量足够（≥3 份病例）时才有效
4. **病历类型插件系统**：工作量最大（9-13 天），如果当前 7 种类型已满足需求，可延期
