## 二、P0-1：修复 `category` 保存链丢失

### 现象
同意书、护理记录等保存后，数据库中 `category` 全部变成默认值 `clinicalRecords`。

### 根因
- 前端 `EmrPreview._saveRecord()` 构建记录对象时未包含 `category`
- 后端 `crud.js` 解构 `req.body` 时未取 `category`
- 后端 `database.saveRecord()` 只在 INSERT 时用 `record.category || 'clinicalRecords'`，UPDATE 路径也没有 category

### 实施步骤

1. **前端 `public/js/components/EmrPreview.js`**
   - 在 `_saveRecord()` 中，拿到 `typeConfig` 后，通过 `store._findCategoryForType(activeType)` 或从 registry 遍历找到所属 category
   - 在构造 `record` 对象时增加 `category: categoryId`

2. **后端 `src/routes/crud.js`**
   - 在 `POST /api/records` 的解构中增加 `category`
   - 调用 `db.saveRecord({ ..., category })` 时传入

3. **后端 `src/services/database.js`**
   - `saveRecord()` 的 UPDATE 路径增加 `category = ?` 的 set clause
   - UPDATE 的 params 中在 `type` 之后加入 `record.category || 'clinicalRecords'`

### 验收方式
1. 在 UI 中生成并保存一份「手术同意书」
2. 查询数据库 `SELECT category, type FROM records ORDER BY createdAt DESC LIMIT 1;`
3. 预期 `category = 'consentForms'`，而非 `clinicalRecords`
4. 编辑已保存记录后再次保存，`category` 保持不变

## 三、P0-2：修复 `content` JSON 列 UPDATE 不更新

### 现象
编辑已保存病历后重新保存，`content` 列仍是旧 JSON，与扁平列数据产生静默漂移。

### 根因
`database.saveRecord()` 的 UPDATE 分支只更新扁平列和 `updatedAt`，不更新 `content`。

### 实施步骤

1. **后端 `src/services/database.js`**
   - 在 `saveRecord()` 中，把 `content` 的生成逻辑提到 `if (existing)` 判断之前
   - UPDATE 的 `setClauses` 增加 `'content = ?'`
   - UPDATE 的 `params` 在 `type` 之后加入 `JSON.stringify(content)`
   - 注意：UPDATE 时 `category` 也要同步更新（与 P0-1 一起处理）

### 建议的 `setClauses` 顺序
```
disease = ?, type = ?, category = ?, content = ?, <data columns...>, updatedAt = ?
```

### 验收方式
1. 保存一份记录，记录其 `content`
2. 修改某字段后再次保存
3. 查询数据库 `SELECT content FROM records WHERE id = ?;`
4. 预期 `content` 中对应字段已更新，且 `updatedAt` 已变化

## 四、P0-3：修复 Prompt 编辑器门控

### 现象
配置器新增的自定义类型 `templateKey` 为 `generic_xxx`，由于 `defaultPrompts.json` 中无对应条目，Prompt 编辑器里根本不显示该类型 Tab。

### 根因
`PromptEditor.js` 多处把 `defaultTemplate.templates[type.templateKey]` 是否存在作为显示/读取的前提。

### 实施步骤

1. **`_buildTabOrder()`**
   - 移除 `if (this._state.defaultTemplate?.templates?.[type.templateKey])` 门控
   - 改为：所有 `enabled !== false` 的 Registry 类型都加入 Tab 列表
   - Tab 顺序仍优先按 Registry 的 `sortOrder`

2. **`_renderTabContent()`**
   - 当 `defaultTemplate.templates[typeKey]` 不存在时，从 Registry 的 `type.fields` 渲染字段表
   - 字段 label 优先用 Registry 的 `field.label`
   - 字段 description 优先用 Registry 的 `field.description`

3. **`_collectCurrentTabData()`**
   - `label` 来源增加 Registry fallback：`typeConfig?.label || defaultType?.name || typeKey`

4. **`_showDiff()`**
   - 同时迭代 `defaultTmpl.templates` 和 Registry 类型
   - 对 Registry 有但 default 没有的类型，也要显示 diff（即全新类型）

### 验收方式
1. 在配置器新增一个自定义类型（会生成 `generic_xxx` templateKey）
2. 打开 `/prompts` 页面
3. 预期能看到该新类型的 Tab，并能编辑其 rolePrompt / outputFormat / userPrompt / 字段描述
4. 保存后重新打开，数据不丢失

## 五、P0-4：修复提示词模板名的特殊文件名风险

### 现象
`sanitizeTemplateName` 没有过滤 `.`，可创建 `.gitignore.json`、`.env.json` 等隐藏/特殊文件。

### 根因
正则 `/[\\/:*?"<>|]/g` 只过滤了路径分隔符和 Windows 非法字符，未过滤 `.`。

### 实施步骤

1. **后端 `src/services/promptTemplates.js`**
   - 修改 `sanitizeTemplateName()`：
     - 使用严格白名单正则 `/^[a-zA-Z0-9\u4e00-\u9fa5_-]+$/`
     - 任何包含 `.`、空格、特殊字符的名称直接抛错
     - 保留保留名检查（`default`）
   - 更新错误提示文案，明确告诉用户允许的字符

2. **（可选）增加长度限制**
   - 限制名称长度不超过 50 字符，避免文件系统异常

### 验收方式
1. 在 Prompt 编辑器尝试保存模板名为 `.gitignore`
2. 预期后端返回 400，提示非法字符
3. 保存正常名称如 `我的自定义模板_v2` 应成功

## 六、P0-5：修复 mock 模式被环境 API key 绕过

### 现象
前端选择离线/mock 模式时，如果服务端 `.env` 配置了对应 provider 的 API key，后端仍会调用真实 API。

### 根因
- 前端用 `__offline__` 判断离线模式，但请求时仍可能传 `provider: openai` 等
- 后端 `api.js` 用 `isMockMode = !apiKey` 判断
- 后端 `ai.js` 的 `hasApiKey()` 会 fallback 到 `process.env` 的 key

### 实施步骤

1. **前端 `public/js/services/api.js`**
   - 在 `getModelConfig()` 中，当 `activeId === '__offline__'` 时，返回固定对象：
     ```js
     return {
       provider: '__offline__',
       model: '',
       apiKey: '',
       baseUrl: '',
     };
     ```
   - 确保所有 AI 调用函数（`chatCompletion`, `chatStream`, `generateRecord` 等）都透传该 `provider`

2. **后端 `src/services/ai.js`**
   - 在 `hasApiKey()` 最前面增加：
     ```js
     if (provider && resolveProvider(provider) === '__offline__') return false;
     ```
   - 在 `callAI()` 和 `streamAI()` 开头，若 `resolveProvider(provider) === '__offline__'`，直接走 mock
   - 或在 `hasApiKey()` 返回 false 即可，因为 `callAI`/`streamAI` 已经检查 `!hasApiKey()` 走 mock

3. **后端 `src/services/ai.js` `resolveProvider()`**
   - 识别 `__offline__` 并原样返回，不要 fallback 到 `openai`

4. **后端 `src/routes/api.js`**
   - 统一端点 `/records/:typeId/generate` 中，mock 判定改为：
     ```js
     const isMockMode = !apiKey || resolveProvider(provider) === '__offline__';
     ```
   - deprecated 端点同样处理

### 验收方式
1. 在 `.env` 配置任意有效的 API key
2. 前端 Settings 选择「离线模式」或 `__offline__`
3. 生成任意病历
4. 预期不走真实 API，返回 mock 数据
5. 服务端日志/网络请求确认无外部调用
