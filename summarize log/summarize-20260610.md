# 项目总结 — 2026-06-10

## 一、已实现的功能

### 1. 本地数据库（SQLite）

- **位置**：`emr-local-v2/data/emr-local.db`
- **表结构**：`patients`（患者）+ `records`（病历，含 51 个字段列 + JSON content 列）
- **支持 7 种病历类型**：首次病程录、主治查房、主任查房、术前小结、术前讨论、手术记录、出院小结
- **前后端双写**：后端 SQLite 持久化 + 前端 IndexedDB 回退，确保离线可用

### 2. AI 生成系统

- **后端**：`src/routes/api.js`（3 个生成端点 + 6 个模板查询端点 + 对话端点）
- **前端**：`EmrPreview.js`（7 标签页切换 + 重新生成 + 保存）
- **多模型支持**：OpenAI / Claude / Gemini / DeepSeek / Ollama（本地）
- **离线模式**：`ai-mock.js` 提供模拟数据，无需 API Key 也能运行
- **流式输出**：对话和首次病程录生成支持 SSE 流式

### 3. 病历生成提示词

- **首次病程录**：9 个字段（chief/hpi/past/exam/lab/diag/workup/diff/plan）
- **主治查房**：5 个字段（summary/diagnosis/analysis/treatment/signed），继承首次病程录内容
- **提示词优化**：分号序号格式、伴发诊断强制列出、字段联动规则

### 4. 对话系统

- 流式 AI 对话，支持通过对话修改病历字段
- 步骤化提示词（4 步流程 + 最小改动原则 + 字段联动规则）
- 返回 JSON 代码块自动合并到病历

## 二、已规划但未实现的功能

### 1. RAG 知识库

- **方案**：在 `src/data/medical-files/` 下按疾病名建目录，存放 MD 格式专业医学文件
- **实现**：新建 `src/services/knowledge.js`，`loadKnowledge(disease)` 按需读取文件内容
- **注入方式**：将知识库内容注入提示词，AI 优先基于文件内容生成
- **字符限制**：8000 字符上限，防超出 LLM 上下文窗口
- **离线兼容**：知识库在路由层加载，在线/离线模式均可使用（但离线 mock 不会真正引用内容）

### 2. 小模型 + RAG 方案

- **核心思路**：知识库提供专业上下文，小模型（Ollama 1.5B-7B）负责读取和整理信息
- **效果预估**：有知识库的小模型（80-85 分）≈ 无知识库的大模型（70 分）
- **分级策略**：有知识库的疾病用小模型（免费），无知识库的回退到大模型
- **完全离线**：Ollama 本地运行 + 本地知识库，零网络依赖

### 3. 记忆系统（自动聚合模板）

- **触发条件**：用户保存病例后，系统检查该疾病是否有 ≥3 份已保存病例
- **聚合方式**：AI 分析共同模式（主诉模式、体征特点、诊断模式、治疗方案）
- **存储位置**：新建 `learned_templates` 表或 `src/data/learned-templates/` 目录
- **模板使用**：下次生成同疾病病例时，优先使用聚合模板（比预置模板更贴近实际）
- **版本管理**：每次聚合生成新版本，保留历史版本，支持用户编辑修正

## 五、关键设计决策

| 决策 | 选择 | 理由 |
|---|---|---|
| 数据库 | SQLite + JSON 列 | 轻量、单机部署、灵活扩展 |
| 离线方案 | Mock + Ollama | 无需网络、本地运行 |
| 知识库格式 | MD 文件按疾病分类 | 简单、易维护、版本控制友好 |
| 模板聚合 | 自动触发（≥3份病例） | 数据驱动、减少人工干预 |
| 模型策略 | 分级（小模型+知识库 / 大模型） | 成本与效果平衡 |
