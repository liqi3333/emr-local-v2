## S1-1：修复 `content` JSON 列 UPDATE 不更新

### 问题
编辑已保存病历后重新保存，`content` 列仍是旧 JSON，与扁平列数据产生静默漂移。

### 涉及文件
- `src/services/database.js`

### 实施步骤

1. 将 `content` 的生成逻辑从 INSERT 分支中提取到 `if (existing)` 判断之前
2. 保持使用 `_buildRecordContentFromRegistry(record, typeConfig)` 优先，fallback 到 `_buildRecordContent(record)`
3. 在 UPDATE 分支的 `setClauses` 中增加 `'content = ?'`
4. 在 UPDATE 的 `params` 中，在 `type` 参数后（或 `category` 参数后）加入 `JSON.stringify(content)`
5. 同步为 S1-2 的 category 更新预留参数位置

### 建议的 UPDATE 参数顺序

```
[disease, type, category, content, <data columns...>, updatedAt, id]
```

### 验收方式

1. 保存一份 firstCourse 记录，记录其 `content`
2. 修改 `chief` 字段后再次保存
3. 执行 `SELECT content FROM records WHERE id = ?;`
4. 预期 `content` 中的 `chief` 已更新，且 `updatedAt` 已变化

## S1-2：后端接收并保存 `category`

### 问题
后端不接收前端传入的 `category`，保存时全部使用默认值 `clinicalRecords`。

### 涉及文件
- `src/routes/crud.js`
- `src/services/database.js`

### 实施步骤

1. **修改 `src/routes/crud.js`**
   - 在 `POST /api/records` 的字段解构中增加 `category`
   - 调用 `db.saveRecord({ ..., category })` 时传入该字段

2. **修改 `src/services/database.js`**
   - 在 UPDATE 分支的 `setClauses` 中增加 `'category = ?'`
   - 在 UPDATE 的 `params` 中，在 `type` 参数后增加 `record.category || 'clinicalRecords'`
   - 确保 INSERT 分支保持 `record.category || 'clinicalRecords'`

### 验收方式

1. 使用 API 工具发送 `POST /api/records`，body 中包含 `category: 'consentForms'`
2. 查询数据库，确认 `category` 为 `consentForms`
3. 不传 category 时，应 fallback 为 `clinicalRecords`

## S1-6：修复前端 API 客户端无条件 `res.json()`

### 问题
无论响应 Content-Type 是什么，前端都调用 `res.json()`。服务端返回 HTML 时会抛语法错误。

### 涉及文件
- `public/js/services/recordTypeApi.js`
- `public/js/db.js`

### 实施步骤

1. 在 `recordTypeApi.js` 的 `request()` 中：
   - 先检查 `res.ok`
   - 如果 `!res.ok`，读取文本并根据 Content-Type 尝试解析 JSON，否则返回文本错误
   - 抛出包含真实 HTTP 状态码和消息的错误
2. 在 `db.js` 的 `_fetch()` 中做同样处理
3. 只在 `res.ok` 且 Content-Type 为 JSON 时才调用 `res.json()`

### 验收方式

1. 让服务端返回一个非 JSON 的 500 错误
2. 前端应抛出包含 HTTP 状态码的错误，而不是 `SyntaxError`

## S3-2：修复 mock 模式被环境 API key 绕过

### 问题
前端选择离线/mock 模式时，如果服务端 `.env` 配置了 API key，后端仍会调用真实 API。

### 涉及文件
- `public/js/services/api.js`
- `src/services/ai.js`
- `src/routes/api.js`

### 实施步骤

1. **前端 `public/js/services/api.js`**
   - 在 `getModelConfig()` 中，当 `activeId === '__offline__'` 时，返回固定对象：
     ```js
     return {
       provider: '__offline__',
       model: '',
       apiKey: '',
       baseUrl: '',
     };
     ```
   - 确保所有 AI 调用函数都透传该 `provider`

2. **后端 `src/services/ai.js`**
   - 修改 `resolveProvider()`：识别 `__offline__` 并原样返回
   - 修改 `hasApiKey()`：最前面增加 `__offline__` 强制返回 false

3. **后端 `src/routes/api.js`**
   - 统一端点 `/records/:typeId/generate` 中：
     ```js
     const isMockMode = !apiKey || resolveProvider(provider) === '__offline__';
     ```
   - deprecated 端点同样处理
   - 引入 `resolveProvider`：`const { resolveProvider } = require('../services/ai');`

### 验收方式

1. 在 `.env` 配置任意有效的 API key
2. 前端 Settings 选择「离线模式」
3. 生成任意病历
4. 预期返回 mock 数据，不走真实 API
5. 确认无外部网络请求
