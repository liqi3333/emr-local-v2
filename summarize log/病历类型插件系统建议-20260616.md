# 病历类型插件系统建议

三重身份审视：资深程序员 × 临床医学专家 × 外科高级主任医师

---

## 一、作为资深程序员

### 做得好的

- Registry 作为唯一数据源，前端/后端/AI/Mock 统一消费，是正确决策
- 3 层提示词回退机制（模板→自动生成→报错）保证新类型开箱即用
- Context Dependencies 显式声明依赖图，比硬编码更灵活

### 需要改进的

| 问题 | 建议 |
|------|------|
| 60 列扁平 schema | 每个记录行有 50+ 个空字符串列。改为 JSON content 为主存储，保留少量索引列（patientId, disease, type, createdAt） |
| Prompt 编辑器与 Registry 脱节 | 编辑器应从 Registry 动态读取类型列表，去掉硬编码 TAB_ORDER |
| 两套字段描述 | Registry 简短版 vs 模板详细版，应统一为一个权威源 |
| `_buildRecordContent` 13 个 case | 已有 Registry 路径，旧 switch/case 应标记 DEPRECATED 并逐步移除 |
| 无 undo/redo | `contentEditable` 编辑无历史记录，建议加 local undo stack |

## 二、作为临床医学专家

### 临床逻辑缺口

| 缺口 | 影响 | 建议 |
|------|------|------|
| **诊断一致性无校验** | 首次病程写了「腹股沟疝」，出院小结可能写成「腹股沟斜疝」，系统不检测 | 跨类型诊断字段一致性检查 |
| **鉴别诊断无模板引导** | `diff` 字段描述太简略，年轻医生不知道该鉴别哪些疾病 | 按疾病提供鉴别诊断要点列表 |
| **缺少「必要字段」标记** | 所有字段平等，但「主诉」「初步诊断」是必须写的，「签名」是必须留空的 | 字段增加 `required` 属性 |
| **病程记录缺少时间线** | 各记录独立存储，无法看到「入院→检查→手术→出院」的完整时间线 | 增加时间线视图或 Timeline 组件 |
| **无医嘱/处方集成** | 治疗计划是纯文本，无法结构化执行 | 远期可考虑医嘱模块 |

### 外科医师最关心的 3 件事

1. **手术记录的准确性** — 术中发现、手术经过应该能引用术前讨论的结论（contextDependencies 已支持，但提示词模板没有引导 AI 这样做）
2. **同意书的法律效力** — 风险告知必须完整，当前模板只有「腹股沟疝」一个疾病，覆盖不足
3. **护理记录的连续性** — 护理评估→护理计划→护理记录单应该有数据传递（contextDependencies 有，但占位符缺失）

## 三、作为外科高级主任医师

### 科室管理维度

| 维度 | 现状 | 期望 |
|------|------|------|
| **病历质量控制** | 无自动质控 | 应有必填字段检查、诊断一致性检查、字数/格式检查 |
| **模板标准化** | 7 个老类型有详细模板，6 个新类型只有 1 个疾病 | 所有同意书/护理记录应覆盖常见病种 |
| **教学功能** | 无 | 新入科医生可以参考模板学习规范病历书写 |
| **医疗纠纷防范** | 同意书模板不够全面 | 应覆盖所有常见手术的风险告知要点 |
| **效率** | 手动点击每个标签生成 | 应支持「一键生成全套病历」（按 contextDependencies 顺序批量生成） |

---

## 四、优先级建议

### 短期（现有系统完善）

1. 修复 Prompt 编辑器与 Registry 联动（提示词插件问题-待解决 问题 1-5）
2. 补全上下文占位符（`{{nursingAssessmentContext}}` 等）
3. 扩充同意书/护理记录的疾病模板覆盖

### 中期（临床功能增强）

4. 字段增加 `required` 属性 + 保存时校验
5. 跨类型诊断一致性检查
6. 「一键生成全套病历」功能

### 长期（架构升级）

7. 数据库 schema 优化（JSON content 为主）
8. 时间线视图
9. 打印/PDF 导出
10. 病历质控报告
