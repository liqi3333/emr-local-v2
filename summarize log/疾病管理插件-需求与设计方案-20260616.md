# 疾病目录管理插件 — 需求与设计方案

> 日期：2026-06-16
> 项目：emr-local-v2

---

## 一、需求描述

### 1.1 左侧疾病树管理

需要在左侧疾病栏实现一个管理插件，支持：

- **添加一级分类**（如"呼吸科"、"心血管科"）
- **添加二级疾病**（如"肺炎"、"肺癌"）
- **删除现有疾病和分类**
- **修改疾病和分类名称**
- **修改分类的图标、颜色样式**

### 1.2 新疾病的功能完整性

新添加的一级标签和二级标签疾病，必须能够使用**所有现有病历功能**：

- 首次病程录
- 主治查房
- 主任查房
- 术前小结
- 术前讨论
- 手术记录
- 出院小结
- 手术同意书、输血同意书、麻醉同意书
- 护理评估、护理计划、护理记录单

### 1.3 右侧字段管理

右侧一级和二级标签栏下的所有字段，需要支持**增加、删除、启用/禁用、排序**功能。

## 二、方案分析

### 2.1 核心结论：两个独立模块

```
左侧疾病树                   右侧病历类型系统
───────                     ───────────────
疾病增删                     字段增删
（本次开发）                 （已有功能，无需开发）
```

两者是**独立的、互不干扰**的模块：

| 模块 | 决定什么 | 状态 |
|------|---------|------|
| 左侧疾病树 | "给什么病写病历"（传给 AI 的疾病名） | **本次开发** |
| 右侧标签栏 + 字段 | "写什么类型的病历、包含哪些字段"（Registry 管理） | **已有功能**，无需改动 |

### 2.2 为什么新疾病不需要额外开发就能用

所有病历都是 **AI 实时生成** 的，疾病名只是传给 AI 的一个参数：

```
用户点"肝癌" → 传给 AI: "请为肝癌患者生成首次病程录"
                                          ↑
                               疾病名只是 prompt 里的变量
```

**在线模式（有 API Key）**：AI 根据疾病名推理生成内容，不依赖离线模板，**所有 13 种病历类型全部可用**。

**离线模式（Mock，无 API Key）**：系统尝试从 `templates.js` 查找预置模板。找不到时，使用通用 Mock 生成器输出占位内容，不会崩溃。

### 2.3 右侧字段管理已有功能

访问 `/record-types` 页面，通过三栏配置器可以：

- 禁用/启用任意字段
- 调整字段顺序（上移/下移）
- 编辑字段的 label 和描述
- 新增自定义字段
- 禁用整个病历类型

这些设置影响所有疾病，不分新旧。

## 三、技术方案

### 3.1 架构

```
浏览器                         服务器
─────                         ─────
DiseaseManager.js  ──PUT────→  /api/diseases  ──→ SQLite settings 表
  (增删改 UI)                   (保存)

DiseaseTree.js     ←─GET────  /api/diseases  ←── SQLite settings 表
  (显示疾病树)                  (读取)

diseases.js
  (加载层：从 API 获取,
   失败时用本地兜底数据)
```

### 3.2 数据格式

沿用现有 `DISEASE_CATEGORIES` 结构，不做改动：

```javascript
[
  {
    name: '腹部急症',       // 分类名称
    icon: '🔥',             // 图标
    color: '#fee2e2',       // 标记色
    textColor: '#b91c1c',   // 文字色
    bgColor: '#fef2f2',     // 背景色
    diseases: ['急性阑尾炎', '急性胆囊炎', ...]  // 疾病列表
  }
]
```

### 3.3 存储方式

存在 SQLite `settings` 表中（key: `disease_categories`），与病历类型注册表（`record_registry`）同模式。

### 3.4 兜底策略

后端 `diseases.js` 中硬编码 10 个默认分类，第一次访问或数据库中没有数据时自动返回。前端 API 请求失败时使用本地 `diseases.js` 中的兜底数据，保证不崩溃。

| 文件 | 原因 |
|------|------|
| `src/data/templates.js` | 离线模板，新疾病没有不影响在线 AI 生成 |

| 右侧字段系统 | 已有 `/record-types` 管理页面，无需改动 |

## 六、操作流程

### 6.1 用户使用流程

1. 点击顶部 **"🏥 疾病"** 按钮
2. 打开 `/diseases` 管理页面
3. 在管理页面中：
   - 点击 **"+ 添加一级分类"** 新建分类
   - 在分类下点击 **"+ 添加疾病"** 新增疾病
   - 点击疾病或分类旁的 **"🗑"** 删除
   - 直接编辑名称、图标、颜色
   - 点击 **"💾 保存"** 写入数据库
4. 回到主界面，左侧疾病树自动显示新内容
5. 点击新疾病，AI 正常生成所有病历类型

### 6.2 开发实施步骤

1. 创建 `src/routes/diseases.js`（后端 API）
2. 创建 `public/js/services/diseaseApi.js`（前端 API 客户端）
3. 创建 `public/js/components/DiseaseManager.js`（管理界面）
4. 创建 `public/diseases.html`（管理页面）
5. 修改 `server.js`（注册路由）
6. 修改 `public/js/data/diseases.js`（动态加载）
7. 修改 `public/js/components/DiseaseTree.js`（使用动态数据）
8. 修改 `public/index.html` + `public/js/app.js`（加按钮）

## 七、风险与注意事项

| 风险 | 说明 | 应对 |
|------|------|------|
| 离线模式新疾病无模板 | Mock 数据质量一般 | 建议使用在线 AI 模式；不影响功能完整性 |
| 管理页面兼容性 | 需保持与现有设计系统一致 | 复用 `style.css` 的 CSS 变量 |
| 数据迁移 | 已有疾病数据迁移？ | 已有记录的数据不受影响，疾病名只是字符串 |
