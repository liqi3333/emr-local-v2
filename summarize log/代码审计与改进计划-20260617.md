## 第一优先级：数据完整性 Bug（必须立即修复）

> 这些 Bug 正在导致数据存储错误，影响已有数据

### B1: category 永远存为 clinicalRecords

- **位置**：`src/routes/crud.js:130-145`
- **问题**：POST /api/records 解构 `req.body` 时未包含 `category`，导致 `database.js:300` 中 `record.category || 'clinicalRecords'` 永远取默认值
- **影响**：同意书（surgeryConsent/bloodTransfusionConsent/anesthesiaConsent）和护理记录（nursingAssessment/nursingPlan/nursingRecordSheet）全部存为 `clinicalRecords` 分类，前端无法按分类筛选
- **修复**：`crud.js` 解构加 `category`，传递给 `db.saveRecord()`

### B2: content JSON 更新后不刷新

- **位置**：`src/services/database.js:278-291`
- **问题**：UPDATE 语句的 SET 子句包含 `disease`、`type`、77 个数据列、`updatedAt`，但没有 `content = ?`
- **影响**：编辑保存后，`content` 列保持 INSERT 时的快照值，与实际数据不一致。预览/导出可能显示旧数据
- **修复**：UPDATE 语句加 `content = JSON.stringify(this._buildRecordContentFromRegistry(record, typeConfig))`

### B3: _buildRecordContentFromRegistry 是死代码

- **位置**：`src/services/database.js:274-276`
- **问题**：`saveRecord(record, typeConfig)` 中 `typeConfig` 从 `crud.js` 调用时永远是 `undefined`（第二个参数未传递），导致走遗留 switch/case（:326-456），Registry 自定义字段不生效
- **影响**：用户通过 RecordTypeManager 禁用字段后，`content` JSON 仍包含被禁用的字段
- **修复**：`crud.js` 查 Registry 获取 `typeConfig` 并传递

### B4: RECORD_DATA_COLUMNS 与 newColumns 重复

- **位置**：`src/services/database.js:21-38`（主数组）vs `:134-142`（ALTER 数组）
- **问题**：31 个同意书/护理记录列名在两个地方重复定义。新增列只改主数组而忘记 ALTER 数组，会导致旧数据库启动时报 `no such column`
- **影响**：维护负担，容易遗漏导致旧数据库崩溃
- **修复**：`newColumns` 从 `RECORD_DATA_COLUMNS` 派生，消除重复

## 第二优先级：架构债务（影响可维护性）

> 不会立即崩溃，但每次新增功能都要付出额外代价

### A1: crud.js 60 字段手动解构+重建

- **位置**：`src/routes/crud.js:130-239`
- **问题**：60 个字段逐个解构（130-145），再逐个重建传递给 `saveRecord`（157-239）。新增字段需要同步修改 `database.js`、`crud.js`、前端 3 处
- **影响**：违反 `RECORD_DATA_COLUMNS` 作为单一来源的原则，维护成本高
- **修复**：用 `RECORD_DATA_COLUMNS` 动态提取，如 `const record = { id, patientId, disease, type, category, ...RECORD_DATA_COLUMNS.reduce((o, c) => ({ ...o, [c]: req.body[c] || '' }), {}) }`

### A2: 6 个模板端点复制粘贴

- **位置**：`src/routes/api.js:324-448`
- **问题**：`/templates/attending/:disease`、`/templates/chief/:disease` 等 6 个端点，每个 ~18 行，结构完全相同，只改函数名
- **影响**：~100 行重复代码，新增类型要加新端点
- **修复**：合并为 `GET /templates/:templateKey/:disease`，用映射表 `{ attending: getAttendingTemplate, ... }`

### A3: 7 个废弃生成端点重复

- **位置**：`src/routes/api.js:199-256`
- **问题**：7 个 `DEPRECATED` 端点（`/emr/generate`、`/attending/generate` 等）与统一端点（`:88`）逻辑重复，且数组归一化行为不一致（:245 用 `JSON.stringify`，:172-174 用 `Object.values().join`）
- **影响**：同一功能两条代码路径，行为不一致导致难以排查问题
- **修复**：废弃端点改为代理到统一端点，删除重复逻辑

### A4: 前端 13 个 legacy 函数仅 DiseaseTree.js 在用

- **位置**：`public/js/services/api.js:121-484`
- **问题**：7 个 `generate*` + 6 个 `get*Template` 函数，共 ~360 行，全部复制粘贴。grep 确认仅 `DiseaseTree.js` 调用
- **影响**：360 行死代码拖累维护，新增类型要改多处
- **修复**：迁移 `DiseaseTree.js` 到统一 `generateRecord()`，删除 legacy 函数

### A5: EmrPreview.js 13 个硬编码订阅

- **位置**：`public/js/components/EmrPreview.js:65-71`
- **问题**：订阅 13 个固定 storeKey（`emrData`、`attendingData` 等）。通过 RecordTypeManager 新增的自定义类型，其 `storeKey` 不会被订阅
- **影响**：新增自定义类型的字段编辑后不会触发重新渲染
- **修复**：改为订阅 `activeType` 变化时重新渲染，或用 `'*'` 通配订阅

### A6: store.js reset() 漏重置 6 个遗留 slot

- **位置**：`public/js/store.js:64-88`
- **问题**：`reset()` 的 `keep` 对象只保留 `patients/records/recordRegistry/diseaseCategories`，重置对象重新声明 `emrData: null` 但遗漏 `attendingData`、`chiefData`、`preopData`、`discussionData`、`surgeryData`、`dischargeData`
- **影响**：切换患者后旧患者数据残留，导致显示错误
- **修复**：从 Registry 动态获取所有 `storeKey` 并重置

### A7: RecordTypeManager 每次微编辑都全量 PUT

- **位置**：`public/js/components/RecordTypeManager.js:55`
- **问题**：每次 toggle/move/edit 都调用 `_save()`，即 PUT 整个 Registry。拖动排序 5 个字段 = 5 次 HTTP 请求
- **影响**：操作卡顿，网络浪费
- **修复**：加 debounce（300ms），或将多次修改合并为一次 PUT

## 第三优先级：性能与安全优化

> 当前数据量小不会出问题，但不适合长期运行

### P1: getRecords() 返回 SELECT * 77 列

- **位置**：`src/services/database.js:252-259`
- **问题**：`getRecords(patientId)` 返回 `SELECT *`，每行 77 个数据列 + 6 个元数据列。前端 `EmrPreview.js:302` 在 JS 中过滤 `r.type === currentType`。无分页、无 LIMIT
- **影响**：患者记录多时传输量大，synchronous better-sqlite3 调用阻塞事件循环
- **修复**：加 `type`/`category` 参数支持服务端过滤；列表查询只选 `id, disease, type, category, createdAt`；加分页

### P2: 无 schema 版本管理

- **位置**：`src/services/database.js:133-171`
- **问题**：34+ 个 `ALTER TABLE` 语句在每次启动时执行，每个单独 try/catch，无事务包裹
- **影响**：启动慢（虽然 SQLite 的 ALTER 失败很快），无法追踪哪些迁移已执行，中途断电可能导致 schema 不一致
- **修复**：引入 `schema_version` 表，按版本号执行迁移，用事务包裹

### P3: SQL 标识符插值未转义

- **位置**：`src/services/database.js:145, 281, 293, 306`
- **问题**：列名通过 `${col}`、`${c}` 直接插入 SQL 字符串。当前安全（源为硬编码数组），但如果未来从 Registry 动态生成列名，恶意 key 会导致 SQL 注入
- **影响**：当前无风险，未来有风险
- **修复**：加标识符验证（只允许 `[a-zA-Z0-9_]`）或双引号转义

### P4: 无 type/category 索引

- **位置**：`src/services/database.js:127-130`
- **问题**：现有索引仅覆盖 `patientId`、`disease`、`createdAt`。前端按 `type` 过滤，但无 `type` 索引
- **影响**：当前数据量小影响不大，未来记录多时按类型查询会全表扫描
- **修复**：`CREATE INDEX IF NOT EXISTS idx_records_type ON records(type);` 和 `idx_records_category`

### P5: POST /record-types/reset 无确认

- **位置**：`src/routes/recordTypes.js:224`
- **问题**：POST 一次请求即可清空所有自定义 Registry 配置，无 body 验证
- **影响**：误操作或恶意请求会清空全部配置
- **修复**：要求 body 包含 `{ confirm: true }`
