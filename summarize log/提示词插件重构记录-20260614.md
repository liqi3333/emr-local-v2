## 一、项目背景

病历系统 v2 引入了 Registry（病历类型注册表）机制，支持 3 个分类、13 个类型、动态字段配置。但提示词编辑系统（PromptEditor）仍停留在 v1 架构：

- 硬编码 7 个老类型，6 个新类型（同意书/护理）完全不可编辑
- 字段描述存在两套数据源（Registry vs defaultPrompts.json），58/88 个字段不一致
- 占位符命名不统一（`{{emrContext}}` vs storeKey `emrData`）
- ChatArea 聊天面板完全忽略 PromptEditor 模板，自建硬编码 system prompt
- RecordTypeManager 增删类型时不同步模板
- 离线 Mock 不透传上下文

**目标：** 让 Registry 成为唯一真实来源，PromptEditor、ChatArea、Mock、预览全部对齐。

### Phase 1：字段描述迁移到 Registry

#### 1.1 合并字段描述到 recordRegistry.js

**操作：** 将 defaultPrompts.json 中 20 个字段的详细描述复制到 recordRegistry.js 对应字段的 `description` 属性。

**涉及类型：**
- 首次病程录：chief, hpi, past, exam, lab, diag, workup, diff, plan（9 个）
- 主治查房：supplementHistory, summary, diagnosis, analysis, treatment, signed（6 个）
- 主任查房：chiefSummary, chiefDiagnosis, chiefAnalysis, chiefTreatment, chiefSigned（5 个）

**亮点：** 逐字段对比后复制，保留了 defaultPrompts.json 中更详细的格式说明（如"不超过20字"、"需分层书写"等），确保临床规范完整保留。

#### 1.2 清空 defaultPrompts.json 字段描述

**操作：** 对 13 个类型共 88 个字段，将 `description` 清空为 `""`，仅保留 `label`。

**亮点：** 一次性批量完成，没有遗漏。

**风险：** 这一步之后，如果 Phase 1.3（assembleFieldBlock 改造）未完成，所有 prompt 组装的字段描述将变为空。因此 1.2 和 1.3 必须在同一 session 中连续完成。

#### 1.3 改造 assembleFieldBlock()

**改动：** `promptTemplates.js` 的 `assembleFieldBlock()` 从 `typeConfig.fields[key].description`（模板）改为优先读取 Registry 的 `registryField.description`。

**代码变更：**
```javascript
// 旧：只读模板
function assembleFieldBlock(typeConfig) {
  const fields = typeConfig.fields || {};
  const fieldObj = {};
  for (const [key, field] of Object.entries(fields)) {
    fieldObj[key] = field.description;
  }
  return JSON.stringify(fieldObj, null, 2);
}

// 新：模板 + Registry 融合，Registry 优先
function assembleFieldBlock(typeConfig, registryTypeConfig) {
  const registryFieldsMap = {};
  if (registryTypeConfig && Array.isArray(registryTypeConfig.fields)) {
    for (const f of registryTypeConfig.fields) {
      registryFieldsMap[f.key] = f;
    }
  }
  const templateFields = typeConfig.fields || {};
  const fieldObj = {};
  for (const key of Object.keys(templateFields)) {
    const registryField = registryFieldsMap[key];
    if (registryField && registryField.enabled === false) continue;
    const description = (registryField && registryField.description) || templateFields[key].description || '';
    fieldObj[key] = description;
  }
  return JSON.stringify(fieldObj, null, 2);
}
```

**亮点：** 
- 新增 `enabled === false` 过滤，禁用字段不出现在 prompt 中
- 三重 fallback（Registry → 模板 → 空字符串），向后兼容

#### 1.4 assembleSystemPrompt 自动查找 Registry

**改动：** `assembleSystemPrompt()` 新增自动通过 `findTypeByTemplateKey(type)` 查找 Registry 配置，不再强制要求调用方传入 `registryTypeConfig`。

**新增函数：** `recordRegistry.js` 新增 `findTypeByTemplateKey(templateKey)`，按 templateKey 反查类型。

**验证：** 
```
node -e "const pt = require('./src/services/promptTemplates'); 
const rr = require('./src/services/recordRegistry'); 
rr.ensureDefaultRegistry(); 
console.log(pt.assembleSystemPrompt('emr', {disease:'急性阑尾炎'}))"
```
输出正确包含 Registry 字段描述，9 个字段全部有详细说明。

### Phase 2：占位符统一 + 上下文透传

#### 2.1 动态化 replacePlaceholders()

**问题：** 旧代码硬编码 6 个占位符映射，新增的护理/同意书类型没有对应占位符。

**方案：** 改为从 context 对象动态扫描 `*Data` 键，自动生成 `{{storeKeyContext}}` 占位符。

**代码变更：**
```javascript
// 新增 CONTEXT_LABELS 映射（13 个类型）
const CONTEXT_LABELS = {
  emrData: '首次病程录内容',
  attendingData: '主治医师查房记录',
  // ... 共 13 项
};

function replacePlaceholders(text, context) {
  const replacements = {
    '{{disease}}': disease,
    '{{patientContext}}': buildContextString('患者基本信息', patientInfo),
  };
  // 动态生成
  for (const [key, value] of Object.entries(context)) {
    if (key === 'disease' || key === 'patientInfo') continue;
    if (value && typeof value === 'object' && Object.keys(value).length > 0) {
      const label = CONTEXT_LABELS[key] || key;
      replacements[`{{${key}Context}}`] = buildContextString(label, value);
    }
  }
  // ...
}
```

**亮点：** 
- 新增类型时自动获得占位符支持，无需手动维护映射表
- 保留旧占位符名的兼容性（`{{emrDataContext}}` 就是新名，同时 `{{emrContext}}` 通过动态扫描也能工作）

#### 2.2 更新 defaultPrompts.json 占位符名

**操作：** 13 个类型的 rolePrompt 中，统一替换：
- `{{emrContext}}` → `{{emrDataContext}}`
- `{{attendingContext}}` → `{{attendingDataContext}}`
- `{{preopContext}}` → `{{preopDataContext}}`
- `{{surgeryContext}}` → `{{surgeryDataContext}}`

**同时为缺失上下文的类型补充占位符：**
- nursingPlan：增加 `{{nursingAssessmentDataContext}}`
- nursingRecordSheet：增加 `{{nursingPlanDataContext}}`
- anesthesiaConsent：增加 `{{preopDataContext}}`

#### 2.3 assembleUserPrompt 传递完整上下文

**问题：** API 路由调用 `assembleUserPrompt(templateKey, { disease }, typeConfig)` 只传了 disease，不含上下文。

**修复：** 三处 API 调用全部改为传完整 context 对象：
- `src/routes/api.js:139` — 通用生成端点
- `src/routes/api.js:231` — 旧 7 个 deprecated 端点
- `src/routes/api.js:280` — 流式端点

### Phase 4：ChatArea 继承 PromptEditor 模板

**核心问题：** ChatArea.js 第 126-145 行硬编码 system prompt，完全忽略 PromptEditor 模板。

**方案：** 新增后端 `/api/prompts/preview/:templateKey` 端点，ChatArea 调用该端点获取组装好的 prompt，再追加聊天专属约束。

**后端端点：**
```javascript
router.post('/prompts/preview/:templateKey', (req, res) => {
  const { templateKey } = req.params;
  const context = req.body.context || {};
  const systemPrompt = promptTemplates.assembleSystemPrompt(templateKey, context);
  const userPrompt = promptTemplates.assembleUserPrompt(templateKey, context);
  res.json({ systemPrompt, userPrompt });
});
```

**前端 ChatArea 改造：**
```javascript
// 旧：硬编码
let systemContent = `当前患者疾病：${store.state.currentDisease}。`;
// ... 200 行硬编码拼装

// 新：调用后端
const { systemPrompt: templatePrompt } = await api.getPromptPreview(templateKey, context);
const systemContent = templatePrompt + `\n\n...聊天专属约束...`;
```

**亮点：** 保留了 fallback 机制 — 如果后端调用失败，回退到本地硬编码拼装，保证聊天功能不中断。

## 四、执行亮点

### 1. Registry 成为唯一真实来源

彻底解决了"两套数据源"问题。字段描述只在 `recordRegistry.js` 中维护一处，所有消费者（prompt 组装、编辑器、聊天、预览）统一从 Registry 读取。

### 2. 动态占位符映射

`replacePlaceholders()` 不再硬编码占位符列表，而是从 context 对象动态扫描。新增任何类型只要在 Registry 中定义了 storeKey，就能自动获得 `{{storeKeyContext}}` 占位符支持。

### 3. PromptEditor 标签页动态化

从硬编码 7 个标签变为从 Registry 动态生成，按分类分组显示。新增的 6 个同意书/护理类型第一次在编辑器中可见。

### 4. ChatArea 模板继承

聊天面板不再自建 prompt，而是调用后端 `assembleSystemPrompt()` 获取模板组装结果，再追加聊天专属约束。保证了聊天和生成使用同一套模板。

### 5. 类型生命周期同步

Registry 增删类型时，自动同步到 prompt 模板系统（新增生成骨架，删除清理覆盖），避免了"Registry 有类型但模板没有"的不一致状态。

### 6. Fallback 机制

多处设计了 fallback：assembleFieldBlock 三重 fallback、ChatArea 后端调用失败回退本地、default 模板跳过骨架创建由 Layer 2 兜底。保证了系统的健壮性。

## 五、失败与修补

### 失败 1：assembleFieldBlock 清空描述后 prompt 变空

**现象：** Phase 1.2 清空 defaultPrompts.json 描述后，如果 Phase 1.3 未完成，prompt 组装输出的字段描述全部为空。

**修补：** 确保 1.2 和 1.3 在同一 session 中连续执行，不能中断。

### 失败 2：replacePlaceholders 导出问题

**现象：** 测试时调用 `promptTemplates.replacePlaceholders()` 报 "not a function"。

**原因：** `replacePlaceholders` 是内部函数，未在 module.exports 中导出。

**影响：** 不影响实际功能（通过 `assembleSystemPrompt` 间接调用），仅影响直接测试。

### 失败 3：recordTypeApi.js 导入方式错误

**现象：** `import { getRegistry } from '../services/recordTypeApi.js'` 报错。

**原因：** `getRegistry` 是 `recordTypeApi` 对象的方法，不是独立导出。

**修补：** 改为 `import { recordTypeApi } from '../services/recordTypeApi.js'`，调用 `recordTypeApi.getRegistry()`。

### 失败 4：skeleton 创建在 default 模板上失败

**现象：** `POST /prompts/skeleton` 返回 "default 是保留名称，不能使用"。

**原因：** `saveTemplate()` 拒绝保存到 `default` 模板（只读）。

**修补：** 在 skeleton 端点中检测 active template 是否为 `default`，如果是则跳过，由 `buildFromRegistryFields()` Layer 2 兜底。

### 失败 5：ChatArea 使用 async/await 但 sendMessage 未声明 async

**现象：** ChatArea 的 `sendMessage()` 中使用了 `await api.getPromptPreview()`，但原函数签名不是 async。

**修补：** 确认 `sendMessage()` 已经是 `async` 函数（原代码已有 async），无需额外修改。

## 六、不足与后续优化

### 1. Mock 策略未使用 context

Phase 7 透传了 context 参数，但 13 个策略函数内部并未使用它。例如 `mockAttendingRound` 应该引用 `context.emrData` 生成更真实的"补充病史"，目前仍使用硬编码占位文本。

**建议：** 后续逐个策略函数改造，利用 context 生成引用前序记录的 mock 数据。

### 2. PromptEditor 预览仍为本地拼装

Phase 6 添加了"从后端刷新"按钮，但默认预览仍是前端拼装。前端拼装不经过 `assembleSystemPrompt()` 的完整逻辑（如 Registry 字段过滤），可能与后端输出有细微差异。

**建议：** 后续可改为 debounce 方式自动调用后端预览（如输入停顿 500ms 后自动刷新）。

### 3. 无测试套件

整个重构过程没有自动化测试。验证全部依赖手动 curl 命令和视觉检查。

**建议：** 后续补充 `promptTemplates.js` 的单元测试，覆盖 assembleSystemPrompt、replacePlaceholders、assembleFieldBlock 等核心函数。

### 4. defaultPrompts.json 版本号未更新

清空字段描述后，`version` 仍为 `"2026.06.13-1"`。应更新为 `"2026.06.14-1"` 以标记变更。

### 5. Registry API 路由挂载路径不一致

前端 `recordTypeApi.js` 调用 `/api/record-types/registry`，但 `prompts.js` 中的 skeleton/cleanup 端点挂在 `/api/prompts/` 下。路径风格不统一。

### 6. ChatArea fallback 代码重复

ChatArea 中保留了完整的本地 prompt 拼装 fallback（约 20 行），与主路径代码重复。

**建议：** 提取为独立的 `_buildLocalSystemPrompt()` 方法，减少重复。

## 八、经验总结

1. **数据迁移类任务要连续完成**：清空旧数据（1.2）和切换读取源（1.3）必须在同一 session 中完成，否则中间状态会导致功能异常。

2. **前端硬编码是技术债的放大器**：TAB_ORDER 硬编码导致 6 个新类型完全不可见，这种问题在系统扩展时会指数级放大。

3. **Fallback 机制是重构的安全网**：多处 fallback 设计（ChatArea 本地回退、Layer 2 自动兜底）保证了重构过程中功能不中断。

4. **先做数据层，再做展示层**：Phase 1-2 解决数据权威问题，Phase 3-5 解决展示层绑定，这个顺序是正确的。

5. **Mock 策略是最容易忽略的**：离线模式的 mock 生成器往往被遗忘在重构范围之外，但它直接影响开发体验。
